# llm_brain 完备化路线图

## 1. 文档目的

本文档面向当前代码基线，整理项目如果要从“可运行原型”继续推进到“更完整、可维护、可扩展的 Agent 工程”，下一步应该补齐哪些能力，以及这些能力的优先级、依赖关系和预期产出。

本文档不是问题清单，也不是目标态 PRD，而是基于现状给出的工程推进路线图。

## 2. 当前判断

结合当前源码实现，可以把项目状态概括为：

- 主执行链已经打通。
- 记忆、技能、日志、配置式 MCP 都已经接到主流程。
- 项目具备基本可运行性和一定可追踪性。
- 当前主要短板不在“缺一个入口”，而在“稳定性、恢复能力、检索质量、测试覆盖和协议级扩展能力”。

因此，后续推进应优先补工程完备性，而不是继续堆更多表面能力。

## 3. 路线图总览

建议将后续工作分为四个阶段：

| 阶段 | 目标 | 结果 |
| --- | --- | --- |
| Phase 1 | 补齐执行可靠性 | 让主链从“能跑”提升到“更稳定地跑” |
| Phase 2 | 补齐记忆与能力路由质量 | 让系统更会选、更会记、更可复用 |
| Phase 3 | 补齐可观测性与测试体系 | 让系统更容易排障、验证和迭代 |
| Phase 4 | 补齐协议与产品化能力 | 让系统具备更强扩展性和交付可能 |

## 4. Phase 1：执行可靠性补强

### 4.1 目标

把当前主执行链从“最小可运行”提升为“具备基本工程稳定性”。

### 4.2 建议补齐项

1. 结构化输出约束

- 给特征提取、任务规划、反思输出增加更严格的结构化解析层。
- 减少当前大量依赖 json.loads 和异常兜底的方式。
- 对模型输出错误区分“格式错误”和“语义错误”。

2. 超时与中止机制

- 给单次模型调用设置超时。
- 给单个子任务设置超时。
- 给整次 Agent 调用设置上限时间。
- 支持任务取消，而不是只能等待自然结束。

3. 工具调用异常治理

- 区分工具执行失败、参数不合法、外部依赖不可达等错误类型。
- 给工具异常定义统一返回结构。
- 支持失败后改选工具或退回纯模型处理。

4. 状态恢复能力

- 把中间状态持久化，而不是只写最终 memory。
- 支持从某个子任务索引继续执行。
- 支持中断后恢复，而不是重新从头执行。

### 4.3 完成标准

- 复杂任务执行时出现异常，不会直接把整次链路打断。
- 至少支持“单次调用失败后能给出明确的错误归因”。
- 至少支持“部分状态恢复”或“从中间步骤继续”。

### 4.4 当前已完成

基于当前代码，本阶段已完成以下改造：

- 已完成：为 feature extractor、planner、reflector 新增统一结构化解析层，集中处理 JSON 提取、格式错误和字段校验错误。
- 已完成：主执行链已接入 request_id 级状态快照持久化，关键阶段会把可序列化状态落盘到 state_snapshot_dir。
- 已完成：新增 Phase 1 回归测试，覆盖结构化解析与状态快照落盘。

当前仍未完成的部分：

- 已完成基础版：单次模型调用、单工具调用、整次 Agent 请求都已接入超时控制与逻辑取消返回。
- 已完成基础版：支持按 request_id 发起协作式主动取消，主请求等待、LLM 调用、工具等待和 CLI Ctrl+C 都会响应取消状态。
- 未完成：真正的底层执行终止能力，例如强制中断底层网络调用、模型推理或外部工具进程。
- 已完成基础版：工具调用异常已接入统一分类与安全包装，异常不会直接打断主链。
- 已完成基础版：工具调用失败后会自动排除失败工具，并在无可用工具时降级为无工具路径继续执行。
- 已完成基础版：工具调用失败后的再路由已支持按错误类型分流，参数错误会降级为无工具/补参路径，可重试失败会尝试扩展候选池选择替代工具。
- 已完成基础版：反思重试路径已支持对子任务进行一次重新规划，把失败步骤改写成更小、更安全的替代子任务序列。
- 未完成：跨步骤恢复或更强的替代工具排序仍需继续补强。
- 已完成基础版：支持从指定 request_id 的最新快照或指定快照文件继续执行。
- 已完成基础版：支持列出某个 request_id 的全部快照恢复点，并按最新、序号、阶段名或文件名选择恢复。
- 未完成：基于快照恢复后的更细粒度节点级恢复能力，例如恢复到节点内部更小粒度状态或交互式恢复编排。

## 5. Phase 2：记忆与路由质量补强

### 5.1 目标

让系统更会选技能、更会利用历史记忆，而不是主要依赖简单关键词重叠。

### 5.2 建议补齐项

1. 记忆分层

- 区分会话主记忆、步骤记忆、长期经验记忆、失败案例记忆。
- 给记忆增加类型字段，而不只是统一塞到 interactions 表。

2. 记忆治理

- 增加记忆去重策略。
- 增加相似记录合并策略。
- 增加归档、删除、压缩和衰减策略。
- 避免随着使用增长，召回质量持续下降。

3. 记忆质量标签

- 给记忆打上 success、retry、blocked、ask_user 等标签。
- 支持优先召回成功案例，弱化失败噪音。

### 5.4 当前已完成基础版

- 已完成基础版：interactions 已增加 memory_type 和 quality_tags 字段，支持在同一记忆表内区分 session_main、step 等类型。
- 已完成基础版：会话主记忆和步骤记忆已开始写入 pending、success、retry、blocked、ask_user 等质量标签。
- 已完成基础版：retrieve_memory() 的排序已纳入质量标签分值，成功案例会优先于阻断/求助噪音被召回。
- 已完成基础版：精确重复的记忆记录会在 add_memory() 阶段自动合并，累计权重并合并关键词与质量标签，减少重复噪音堆积。
- 未完成：长期经验记忆、失败案例记忆等更明确的独立分层策略。
- 未完成：归档、压缩、衰减和相似记录合并等更强的记忆治理能力。

4. 能力路由升级

- 在关键词路由之上增加向量检索或混合检索。
- 已完成基础版：为工具和 Prompt 技能选择增加“为什么命中”的解释信息。
- 支持失败后重新路由，而不是直接对同一路径 retry。

5. 参数规划与工具预校验

- 已完成基础版：在真正调用工具前做参数有效性检查。
- 已完成基础版：对缺失参数、明显类型错误、城市名/时间格式错误等情况做前置拦截。

### 5.3 完成标准

- 同类任务重复执行时，能更稳定复用有效历史。
- 工具与技能选择结果不再只依赖浅层关键词命中。
- 失败重试时能有“换路”的能力，而不是只会重跑。

## 6. Phase 3：可观测性与测试体系补强

### 6.1 目标

让项目具备工程迭代所需的验证与排障能力。

### 6.2 建议补齐项

1. 结构化日志

- 已完成基础版：文件日志已开始输出稳定的 JSON 结构化事件，覆盖 event_type、stage、request_id、duration_ms 等核心字段。
- 未完成：session_id、指标汇总、日志轮转和更完整的字段统一仍需继续补强。

2. 指标统计

- 已完成基础版：request summary 已开始聚合单次请求总耗时、LLM 调用次数、工具调用次数、重试次数、阻断率、反思失败次数等核心指标。
- 已完成基础版：支持按最近请求列表查看历史摘要与关键指标。
- 未完成：跨 request 的全局聚合看板、历史统计汇总和更细的阶段耗时分布仍需继续补强。

3. request_id 查询能力

- 已完成基础版：新增按 request_id 聚合执行摘要的能力，并在 CLI 中提供 /request_summary 命令。
- 已完成基础版：可从 request_id 反查 snapshot、memory、checkpoint 和最终输出。

4. 测试层次补齐

- 已完成基础版：补 Agent 主链集成测试，覆盖成功完成和阻断退出两条主路径。
- 补 LLM mock 测试。
- 已完成基础版：补 memory schema 迁移测试。
- 已完成基础版：补日志链路测试。
- 已完成基础版：补 CLI 命令测试，覆盖 /request_summary 和普通消息入口。
- 已完成基础版：补错误场景测试中的重试超限路径；JSON 解析失败和工具异常路径已有对应自动化覆盖。

### 6.3 完成标准

- 每次改动后，不再只能靠手工 CLI 验证。
- 关键主链、迁移逻辑、日志链路、失败路径都有最小自动化保障。
- request_id 可以作为真实排障入口，而不只是一个日志前缀。

## 7. Phase 4：协议与产品化能力补强

### 7.1 目标

让系统具备更强的扩展能力、交付能力和外部集成能力。

### 7.2 建议补齐项

1. MCP 真实协议接入

- 已完成基础版 stdio transport 接入。
- 已完成基础版连接复用与显式卸载。
- 已完成基础版工具枚举刷新。
- 处理远程服务异常、超时、握手失败等情况。

2. 配置与部署能力

- 增加环境变量覆盖能力。
- 增加敏感配置隔离方式。
- 增加日志级别配置和日志轮转。
- 区分本地、测试、生产环境配置。

3. 交互层升级

- 增加流式输出。
- 增加任务计划可视化或执行进度视图。
- 增加按 request_id 查看最近执行记录的命令。
- 根据需要增加 Web UI 或 TUI，而不只依赖纯 CLI。

4. 策略层能力

- 增加更细粒度的任务分类与执行策略切换。
- 对不同类型任务选择不同的 planning / reflection 强度。
- 对简单任务直接走轻路径，对复杂任务走完整链路。

### 7.3 完成标准

- 系统不再只是本地原型，而是具备对外扩展和持续演进的基础。

## 8. 推荐优先级

如果必须严格排序，建议顺序如下：

1. 执行可靠性补强
2. 记忆与能力路由质量补强
3. 可观测性与测试体系补强
4. MCP 真实协议接入
5. 配置、部署与交互层完善

原因很简单：

- 在主链稳定性没有补齐之前，继续扩能力只会放大不稳定性。
- 在路由和记忆质量没有提升之前，增加更多工具不会显著提升完成质量。
- 在缺少测试和可观测性的情况下，大改动难以持续推进。

## 10. 结论

当前项目最需要的不是“再加更多能力”，而是把已经接上的主链从原型提升为更稳定、更可恢复、更可验证的工程系统。

后续如果继续推进，最合理的路径是：

先补可靠性，再补检索和记忆质量，再补测试与观测，最后再推进真实 MCP 和产品化能力。

这条路线最符合当前代码现状，也最能避免后续越做越难维护。
