# llm_brain 后续工程路线图

## 1. 文档目的

本文档只保留当前代码基线下仍未完成、但值得继续推进的工程事项。

它不回顾已经落地的能力，也不记录历史完成项，而是作为后续开发的待办路线图，帮助继续补齐稳定性、恢复能力、检索质量、观测深度和产品化能力。

## 2. 当前判断

当前项目已经不是“缺少主链”的阶段，后续主要矛盾集中在以下几个方向：

- 还缺真正可控的底层中止与更细粒度恢复能力。
- 记忆与能力路由仍偏启发式，复杂任务下稳定性和复用率还有提升空间。
- 可观测性已有基础，但跨请求聚合、日志治理和更细粒度阶段指标仍然不足。
- MCP 侧虽然能跑真实 stdio，但离更完整的协议工程化还有距离。
- 配置、部署、交互层还没有达到长期维护和交付友好的状态。

因此，后续推进应继续优先补工程完备性，而不是扩更多表面功能。

## 3. 路线图总览

建议把后续工作收敛为四个阶段：

| 阶段 | 目标 | 结果 |
| --- | --- | --- |
| Phase A | 深化执行可靠性与恢复能力 | 让主链面对中断、失败和恢复时更稳、更可控 |
| Phase B | 提升记忆与能力路由质量 | 让系统更会选、更会记、更会换路 |
| Phase C | 深化观测与测试体系 | 让系统更容易排障、评估和持续重构 |
| Phase D | 完善协议、配置与交互层 | 让系统更适合长期扩展与实际交付 |

## 4. Phase A：执行可靠性与恢复能力深化

### 4.1 目标

把当前主链从“具备基础超时、取消和恢复”继续提升到“面对复杂失败场景时仍然可控”。

### 4.2 待推进项

1. 真正的底层执行终止能力

- 为底层模型调用补更强的中止机制，而不只是在逻辑层返回取消状态。
- 为外部工具或子进程型能力补真正的终止能力。
- 明确超时、取消、依赖异常三类信号在底层执行层的边界。

2. 更细粒度的恢复能力

- 在现有快照恢复之上，支持更小粒度的节点级恢复。
- 允许恢复后重新选择执行路径，而不只是从已有状态继续跑。
- 支持更清晰的“恢复前校验”，避免在不一致状态下盲目恢复。

3. 跨步骤失败后的更强 reroute

- 为跨子任务失败累计建立更明确的策略层规则。
- 允许把前序失败纳入后续子任务选择，而不是只看当前步骤附近的失败历史。
- 对替代工具选择增加更稳定的排序依据，而不只是当前候选扩展。

4. 状态与恢复契约治理

- 在已落地的 schema version 与基础恢复前校验之上，继续补更明确的状态迁移策略。
- 把恢复前校验从结构合法性推进到更细粒度的语义一致性校验。
- 明确快照兼容边界、恢复失败时的回退规则，以及后续 schema 演进策略。

5. 运行时资源生命周期治理

- 明确请求线程池、工具执行线程池和 MCP 长连接的生命周期边界。
- 继续补齐悬挂线程、子进程和失活连接的回收策略。
- 继续为超时、取消和底层资源释放建立更一致的治理模型。

当前已启动的基线：

- 工具执行已增加运行生命周期跟踪。
- 超时或取消后会先做宽限清理；无法及时停止的工具会标记为 detached。
- detached tool 已进入 request_summary、recent_requests、request summary 明细和专门的运行态查询命令，便于排障。
- 后台完成后的 detached tool 运行记录会被回收。
- 快照恢复前校验已补到语义层，开始拒绝 retry/replan/failed_tools 越界和非终态提前写入 final_response 的快照。
- 跨子任务失败累计已开始进入 reroute：同一请求内重复失败到阈值的工具会在后续子任务中被自动降级或排除，备选工具也会按历史失败次数做优先级排序。
- 快照恢复已新增显式 reroute 模式：可从 blocked 或中途中断的恢复点重新规划，而不必强依赖旧 plan 继续执行。
- 历史失败已从简单计数扩展到结构化严重度：不同 error_type 会累计为 severity 分数，开始直接影响当前工具优先级与 reroute 备选排序。
- 高严重度历史现在也进入策略层：超过 severity 阈值的工具会被直接排除，系统会在没有安全候选时主动降级到无工具路径。
- no-tool 降级已开始区分“应 ask-user 补参”和“可 direct-answer 安全收敛”的两类提示，减少在无工具场景下的模糊决策。
- no-tool 与 reroute fallback 信号已开始进入 request 级摘要，便于直接在 recent/request summary 里识别策略降级，而不必再翻原始日志。
- 模型依赖异常在 invoke / resume 入口已补稳定识别逻辑，开始覆盖模块 reload 后异常类身份漂移的场景，避免依赖故障退化成通用错误提示。
- request_failed 快照 extra 里的 error_type / error 已开始回填到 triage 与 request_rollup，入口级依赖故障也能进入跨请求 failure signals 聚合。
- request 级失败聚合已开始区分稳定 source bucket：可把失败来源归一化为 agent_invoke、agent_resume、model、tool_runtime、reflection 等分类，并输出 stage+source 热点、source 分布占比摘要，以及 recent-vs-earlier 趋势，后续可继续沿此方向细化全局指标。

### 4.3 完成标准

- 用户主动取消时，不再只是逻辑层停止，而是尽量能真正终止底层执行。
- 恢复执行不再局限于快照级继续跑，而能在更小粒度上恢复或改路。
- 多次失败后，系统能稳定进入“换路”而不是重复撞同一条路径。
- 快照恢复具备更明确的契约、版本和校验机制，不再依赖隐含状态假设。
- 长时间运行后，请求线程、工具线程和 MCP 连接不会持续累积失控资源。

## 5. Phase B：记忆与能力路由质量提升

### 5.1 目标

让系统在复杂任务下更稳定地复用有效经验，并减少浅层关键词匹配带来的误选。

### 5.2 待推进项

1. 记忆分层继续深化

- 引入更明确的长期经验记忆层。
- 为失败案例建立可单独处理的记忆层或视图。
- 区分“当前会话上下文”和“长期可复用经验”的召回路径。

当前已启动的基线：

- 失败步骤已开始单独写入 failure_case 记忆视图。
- CLI 已支持 failure_memories 查询入口，可把 blocked、ask_user、retry、timeout 等经验从成功经验中分开浏览。

2. 记忆治理能力

- 增加相似记录合并，而不只是精确重复去重。
- 增加归档、压缩、删除与衰减策略。
- 控制长期使用后记忆库膨胀对召回质量的影响。

3. 能力路由升级

- 在现有关键词路由之上补混合检索或向量检索。
- 为 Prompt 技能和工具技能补更稳定的排序信号。
- 把失败历史、任务类型和上下文信息纳入能力选择，而不只看局部关键词重叠。

4. 参数规划与补参策略继续增强

- 在预校验之外，补更明确的缺参归因和补参规划。
- 对多参数工具调用补更系统的参数填充策略。
- 在“应向用户追问”和“应自动推断”之间建立更稳定的决策边界。

### 5.3 完成标准

- 同类任务多次执行时，成功经验能更稳定复用。
- 技能和工具选择结果不再主要依赖浅层关键词重叠。
- 失败后改路、补参和降级路径更稳定，不容易在局部震荡。

## 6. Phase C：观测与测试体系深化

### 6.1 目标

让项目具备长期重构、持续调试和问题定位所需的工程保障。

### 6.2 待推进项

1. 结构化日志继续完善

- 补齐更统一的 session_id、source、outcome 等字段约束。
- 增加日志轮转和日志保留策略。
- 减少不同模块间事件字段定义的漂移。

2. 指标体系继续深化

- 增加跨 request 的全局聚合视图。
- 继续增加阶段耗时分布，而不只看整次请求的总量指标。
- 区分模型、工具、反思、恢复等不同路径的阶段性统计。

当前已启动的基线：

- 已增加最近请求的跨 request 聚合视图，可直接查看状态分布、attention 数和阶段耗时总览。
- request_rollup 已支持按状态、resumed 请求和 attention 请求做筛选。
- request_summary 已补 checkpoint 级阶段耗时分布聚合。
- 当前阶段桶已覆盖 planning、subtask、tool、reflection、request、resume 和 other。

3. 测试继续分层

- 补更完整的 LLM mock 测试，降低对真实 provider 的隐式依赖。
- 补更多模块边界测试，保证后续继续拆分时回归成本可控。
- 为日志、恢复、取消和 reroute 增加更细粒度的失败路径覆盖。

4. 排障入口继续增强

- 在 request_id 之上增加更容易浏览的历史视图。
- 为最近请求、失败请求和恢复请求增加更明确的查询维度。
- 让问题定位不只依赖读原始日志，而能通过聚合摘要快速收敛。

当前已启动的基线：

- request_summary 已补 detached tool 计数与明细。
- recent_requests 已补 detached tool 聚合字段。
- recent_requests 已支持按状态、resumed 请求和 attention 请求做筛选。
- recent_requests 和 request_rollup 已支持 since=30m 这类时间窗口筛选。
- recent_requests 已直接展示 attention 原因摘要，减少再跳转 request_summary 的次数。
- recent_requests 对可解析的 latest_failure_details 已进一步提炼 tool、reason、error_type 和 action 摘要。
- failed_requests 和 resumed_requests 已收敛为 recent 查询能力的语义别名，减少重复命令分支。
- request_rollup 已补 top failure signals 摘要，直接暴露常见 failure stage、tool、reason、error_type 和 action。
- request_rollup 已补 failure combination 热点，直接暴露高频 stage+tool 与 tool+reason 组合。
- CLI 已支持 request_rollup 全局聚合入口。
- CLI 已支持直接查看 tracked tool runs。
- retention 基线已覆盖 logs、snapshots、audit logs 和 memory backups，并提供 retention_status 与 dry-run/apply 清理入口；当前同时支持按时间和按数量上限裁剪，并已补低频自动 prune 节流。
- retention_status 现已同时展示最近一次自动 prune 的执行摘要，以及最近一次被跳过时的原因，便于区分 disabled 与 throttled。
- retention 已补 max_bytes 容量上限控制，开始覆盖“按总占用体积裁剪”的场景。

5. 数据保留与清理策略

- 为日志、快照、审计记录和 memory backup 增加 retention 策略。
- 增加归档、轮转、清理和容量上限控制。
- 避免长期运行后磁盘占用和历史噪音持续膨胀。

### 6.3 完成标准

- 每次结构改动后，都能通过更细粒度自动化回归快速判断是否破坏边界。
- 排障时不再主要依赖手工翻日志，而能通过 request、阶段和指标聚合快速定位。
- 日志和指标可以支撑长期运行，而不是只适合短期调试。
- 日志、快照、审计和备份目录具备明确的保留与清理规则。

## 7. Phase D：协议、配置与交互层完善

### 7.1 目标

让系统具备更强的协议扩展性、部署友好性和用户交互体验。

### 7.2 待推进项

1. MCP 工程化继续推进

- 补远程服务异常、超时、握手失败等更完整的失败治理。
- 增加更细粒度连接健康检查。
- 继续补资源同步、订阅刷新或其他更完整的协议能力。

2. 配置与部署能力

- 增加环境变量覆盖能力。
- 增加配置 schema 校验和启动期错误诊断。
- 增加敏感配置隔离策略。
- 区分本地、测试、生产等环境配置。
- 增加日志级别与运行模式配置。

3. 交互层升级

- 增加流式输出。
- 增加任务计划或执行进度视图。
- 增加更适合排障的近期执行浏览命令。
- 视需要引入 Web UI 或 TUI，而不只依赖当前 CLI。

4. 策略层能力

- 按任务类型切换不同的 planning / reflection 强度。
- 为简单任务提供更轻路径，为复杂任务提供更强路径。
- 避免所有任务都走同一套固定执行强度。
- 增加“思考模式”开关：默认模式保持记忆召回的全关键词严格匹配；思考模式再放宽召回阈值（例如允许更低 overlap_ratio 或更小命中数），用于探索型问题。

5. 安全与权限边界

- 为工具调用、MCP 远端能力和终端命令执行建立更统一的权限边界。
- 补日志脱敏、敏感字段屏蔽和 secret redaction。
- 明确本地工具、MCP 工具和高风险能力之间的 trust boundary。

### 7.3 完成标准

- MCP 不再只是“能接 stdio”，而是具备更稳定的连接治理能力。
- 项目配置可以适配不同运行环境，而不需要频繁手改文件。
- 用户交互不再局限于最基础命令行问答。
- 工具与日志层面的安全边界更清晰，默认行为更可控。

## 8. 推荐优先级

如果必须继续严格排序，建议顺序如下：

1. 执行可靠性与恢复能力深化
2. 记忆与能力路由质量提升
3. 观测与测试体系深化
4. 协议、配置与交互层完善

原因很简单：

- 在底层中止、恢复和 reroute 还不够强的时候，继续扩功能只会放大不稳定性。
- 在路由和记忆质量没有继续提升之前，更多工具和能力不会稳定转化为更高完成质量。
- 在完成 A 和 B 之后，再继续深化全局指标和长期日志治理，投入产出会更高。

## 9. 建议拆分成的实际任务包

为了便于后续逐步实施，可以继续拆成以下任务包：

### 9.1 任务包 A：深层执行中止与恢复

- 增加真正的底层执行终止能力
- 增加节点级或更细粒度恢复能力
- 增加恢复前一致性校验
- 增加跨步骤失败后的更强 reroute 策略
- 增加快照 schema version、状态迁移和恢复兼容策略
- 增加请求线程、工具线程和 MCP 连接的生命周期治理

### 9.2 任务包 B：长期记忆治理

- 增加长期经验记忆层
- 增加失败案例视图或专用记忆层
- 增加相似记录合并
- 增加归档、压缩、删除和衰减策略

### 9.3 任务包 C：混合检索与能力选择升级

- 引入混合检索或向量检索
- 增加更稳定的能力排序信号
- 增加失败历史参与的能力选择
- 增加更系统的补参与参数规划

### 9.4 任务包 D：观测与测试深化

- 增加更完整的 LLM mock 测试
- 增加跨 request 聚合指标
- 增加阶段耗时分布统计
- 增加更细粒度失败路径覆盖
- 增加更适合排障的查询视图
- 增加日志、快照、审计和备份目录的 retention / 清理策略

### 9.5 任务包 E：协议与交互产品化

- 增强 MCP 连接治理与协议能力
- 增加环境变量与多环境配置支持
- 增加配置 schema 校验与启动期诊断
- 增加日志轮转与运行级别配置
- 增加流式输出、执行进度视图和更友好的交互层

### 9.6 任务包 F：安全与权限边界治理

- 增加工具与 MCP 调用的统一权限边界
- 增加日志脱敏与 secret redaction
- 增加高风险命令、敏感路径和远端能力的分级控制

### 9.7 任务包 G：能力质量评估基线

- 建立固定任务集评估技能选择、工具命中、补参成功和 reroute 成功率
- 为路由、记忆召回和参数规划建立可重复比较的基线数据
- 避免后续做混合检索或排序升级时只能靠主观判断效果

## 10. 结论

当前项目后续最值得做的，不是再补一批孤立能力，而是继续把可靠性、恢复能力、检索质量、观测深度和协议工程化补齐。

如果继续推进，最合理的路径仍然是：

先把深层执行可靠性和恢复能力补强，再把记忆与能力路由做实，然后继续深化观测与测试，最后再推进更完整的协议、配置和交互层能力。

这条路线最符合当前代码现状，也最能避免后续继续扩展时重新回到“能跑但难维护”的状态。
