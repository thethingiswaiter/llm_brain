# llm_brain 终端交互体验路线图

## 1. 文档目的

进度同步（2026-04-04）：

- 已完成：4.1.3、4.1.4、4.2.1、4.2.2、4.2.3、4.2.4、6.1、6.2、6.3。
- 部分完成：4.1.1、4.1.2、4.3.1、4.3.2、4.3.3、4.3.4、4.4.1、4.4.3、4.4.4，当前已补 /history [limit] 压缩视图、request_summary 内的子任务执行视图、工具反馈视图、常用动作预设命令、输入建议/最近命令历史、少量安全自然语言动作映射，以及统一的 Request Strip 摘要条；默认终端入口已切到 Textual 组件工作台，并已有固定 status、Current Request、Action Context、Recent History 三类区域。
- 待继续：把 Phase 4 里的高频动作、补全和选择逻辑进一步从命令输出迁入 Textual 组件本身。

本文档用于重新聚焦 llm_brain 的下一阶段建设重点，并且明确在切换到 Textual 组件化终端之后，后续迭代应该落在哪些真实文件和组件上。

当前最需要补强的，不是继续堆叠更多底层能力，而是把终端里的交流体验、信息查看体验和命令执行体验整理得更清晰、更顺手，让它真正适合高频使用。

因此，这份路线图主要围绕以下目标展开：

- 让用户在终端中与 Agent 交流时，更容易看清上下文、状态、结果和风险。
- 让常用命令更容易发现、更容易输入、更容易组合，不必频繁记忆长命令。
- 让运行中信息的展示从“日志堆叠”变成“分层视图”，减少阅读负担。
- 让终端逐步从“调试入口”演进为“主工作界面”。

### 1.1 当前实现落点

当前终端相关路径已经变化，后续 roadmap 应以这些真实落点为准：

- main.py：终端总入口。默认优先启动 Textual TUI。
- app/cli/textual_app.py：当前主终端实现。负责 Textual 布局、状态区、对话区、右侧工作台侧栏，以及后续组件化交互。
- app/cli/commands.py：命令解析与动作执行层。当前仍是 Textual 和旧行式 CLI 共用的核心行为层。
- app/cli/main.py：旧行式 CLI 入口。现在主要作为回退路径和兼容层存在，不再是主体验承载点。
- app/cli/terminal_ui.py：旧行式 CLI 的纯文本渲染层。职责已收缩，不再承担富文本或主工作台建设。

这意味着后续凡是涉及“终端主界面”“固定区域”“交互式选择”“动作面板”“状态栏”的工作，默认都应优先落在 app/cli/textual_app.py，而不是继续向 app/cli/main.py 和 app/cli/terminal_ui.py 叠加能力。

## 2. 当前问题判断

结合当前 Textual 主入口和命令层结构，终端侧已经具备基础能力，但仍存在几个明显问题：

### 2.1 信息可见性不足

- 当前输出更偏向事件流和结果流，缺少针对“当前正在发生什么”的稳定视图。
- 一次请求中虽然有 request summary、checkpoint、rollup 等能力，但用户需要主动记命令去查，交流过程本身并不自然。
- 普通回复、阶段状态、失败提示、建议操作还没有形成统一层次，重要信息容易被淹没。

### 2.2 命令执行门槛偏高

- 命令数量已经不少，但仍偏向“知道命令的人才能用”。
- 参数形式较长，很多操作要记完整命令格式，缺少快捷入口和默认行为。
- 命令之间缺少串联，查看信息后往往还要手动再敲下一条命令，不能顺滑继续动作。

### 2.3 交流过程仍然偏工程视角

- 终端现在更像运行台和调试台，而不是面向沟通的工作台。
- 用户在对话时希望同时看到：原始问题、重述意图、当前子任务、工具行为、最终回答，但这些内容还没有形成统一展示模型。
- 对于非明确任务交流，虽然流程上已经做了轻量化，但终端展示上还没有对应的轻量化表达。

### 2.4 终端视图层仍不稳定

- 入口已经切到 Textual，但很多高价值动作仍然停留在命令输出文本里，没有真正进入 Textual 组件。
- 当前 Textual 侧栏已经有固定区域，但仍主要展示文本摘要，还没有发展成可直接操作的组件工作台。
- 文档和心智模型如果还按“旧 CLI 增强”来描述，就会继续把工作做错路径。

## 3. 总体设计方向

终端体验后续应围绕四个核心方向展开：

### 3.1 对话优先

- 终端主界面首先服务于交流，而不是服务于日志。
- 用户输入后，界面应优先展示“理解结果”和“当前动作”，日志与明细下沉到二级视图。

### 3.2 视图分层

- 把输出分成主回复区、过程状态区、附加信息区、可执行操作区。
- 默认只展示当前最重要的信息，详细信息按需展开。

### 3.3 命令最小化记忆成本

- 用户不应该依赖记忆完整命令才能高效使用系统。
- 高频操作需要有快捷别名、上下文动作、交互式选择和默认参数策略。

### 3.4 组件化工作台优先

- 新的终端建设应优先沉淀为 Textual 的固定组件，而不是继续往输出流里追加提示文本。
- 命令层继续作为动作后端存在，但主体验应逐步从“输入命令得到文本”转成“操作组件触发命令后端”。
- 路线图中的“状态栏、历史、动作入口、子任务、工具反馈”都应默认被理解为 Textual 工作台组件，而不是旧 CLI 的格式优化。

## 4. 路线拆分

## 4.1 Phase 1: 终端信息层重构

目标：先把“看信息”这件事做好，让 Textual 工作台里的稳定区域和旧行式回退输出共享同一套信息模型。

### 4.1.1 建立统一输出层级

当前状态：部分完成。

- 重新定义终端输出类型：用户输入、意图重述、系统状态、工具动作、最终回答、风险提示、建议动作。
- 每种类型采用固定视觉样式，避免所有内容看起来都一样。
- 主回复必须始终最醒目，过程信息不能压过最终回答。

已落地部分：

- 普通回复与 request_summary 已稳定分成 Summary、Conversation Context、Blocked Reason、Tool Feedback、Subtask View、Final Response、Next Actions 等块状输出。
- 旧行式 CLI 现已显式输出 Summary、Conversation Context、Risk、Request Assets、Final Response、Next Actions 等标题，文本模式具备稳定分区。
- Textual 主界面已经接管终端入口，并开始把这些信息沉淀到固定组件区域，而不是继续堆叠输出。

### 4.1.2 增加请求过程摘要条

当前状态：部分完成。

- 每次请求开始后，固定显示当前 request 的摘要条。
- 摘要条至少包含：request id、当前阶段、是否 lite mode、当前子任务索引、工具调用数、耗时。
- 请求结束后，摘要条自动转为“完成态”或“阻塞态”。

已落地部分：

- 旧行式 CLI 与 Textual 终端都需要保留统一的 Request Strip 信息模型，展示 request id、latest stage、mode、progress、tool_calls、total_ms。
- 当前 Textual 已有固定 status 区，但 Request Strip 语义还没有完全拆成独立组件，后续应把 request 摘要从文本拼接进一步收口成结构化区域。

### 4.1.3 增加意图理解展示

当前状态：已完成。

- 用户发出请求后，在执行前展示“原始输入”和“意图重述”的对照。
- 如果命中了轻聊天流程，应明确提示“已走轻量交流路径”。
- 如果进入任务路径，应展示“当前按任务处理”的判断结果。

### 4.1.4 增加失败与阻塞的可读化输出

当前状态：已完成。

- 阻塞时不只输出错误文本，还要给出“为什么阻塞”“下一步最建议做什么”。
- 失败原因要从原始日志式细节转成用户可读摘要。
- 对 dependency unavailable、tool timeout、missing parameter 等常见问题做模板化展示。

## 4.2 Phase 2: 命令交互重构

目标：降低命令使用门槛，让常见操作更直接。

### 4.2.1 命令别名与短命令

当前状态：已完成。

- 为高频命令补充简写，例如 summary、recent、rollup、snapshots、resume、mcp 等。
- 保留原命令兼容性，但主帮助优先展示短命令。
- 命令别名需要统一规范，避免一个功能多个风格混用。

### 4.2.2 参数默认化

当前状态：已完成。

- 对最近一次 request 相关命令，默认作用于 last_request_id。
- 对最近一次失败 request，允许直接执行恢复、查看快照、查看失败摘要。
- 对需要 request_id 的高频命令，减少必须手输完整 id 的场景。

### 4.2.3 交互式命令选择

当前状态：已完成。

- 对列表型命令结果提供“选择后继续操作”的模式。
- 例如 recent_requests 结果中，用户可直接选择某一项继续 request_summary、list_snapshots、resume_snapshot。
- 让终端命令从“一次性文本输入”进化为“浏览 + 选择 + 执行”的交互。

### 4.2.4 上下文动作面板

当前状态：已完成。

- 在关键输出后追加一组下一步建议动作。
- 例如请求完成后显示：查看摘要、查看快照、查看工具运行、继续追问。
- 请求阻塞后显示：补充参数、恢复快照、查看失败记忆。

## 4.3 Phase 3: Textual 工作台组件

目标：让终端交流从单轮输出变成持续工作的组件化界面。

### 4.3.1 会话侧栏或顶部概览

当前状态：部分完成。

- 显示当前 session、最近 request、模型、MCP 数量、lite/full 模式、活跃工具数。
- 把这些信息稳定保留在终端可见区，而不是每次都散落在输出文本中。

已落地部分：

- 这类顶部概览能力应迁入 Textual 的固定状态区，而不是继续依赖输入前打印。
- 当前的 session、last_request、model、MCP、selection、latest request status 等摘要，后续都应统一收拢到 Textual 状态栏。
- Textual 当前已具备固定状态区和右侧工作台侧栏，开始承载 Current Request、Action Context、Recent History 三类稳定信息区。

后续落点：

- app/cli/textual_app.py 中的 status 区应继续拆成更明确的 request / model / runtime 段，而不是保持单行摘要文本。
- 旧行式 CLI 不再继续扩展顶部概览，只保留必要回退能力。

### 4.3.2 对话历史压缩视图

当前状态：部分完成。

- 支持只看最近几轮“用户输入 + 意图重述 + 最终回答”。
- 支持把详细阶段日志折叠到次级视图。
- 让终端交流更像聊天界面，而不是日志窗口。

已落地部分：

- 已新增 /history [limit]，可查看最近请求的输入、意图重述、最终回复、模式与状态。
- /history 结果已接入序号选择，可继续配合 /pick 执行摘要、快照或恢复动作。

后续落点：

- Recent History 应从“侧栏中的多行文本”继续演进成可聚焦、可选择的 Textual 列表组件。
- /history 及 /pick 的能力应逐步下沉为组件选择行为，而不是停留在文本命令链路。

### 4.3.3 子任务执行视图

当前状态：部分完成。

- 当前存在规划、子任务、工具调用、反思等结构，但用户只能间接看到。
- 应提供一块明确的子任务视图，展示当前计划、进行中步骤、已完成步骤。
- 对 lite mode 请求，则展示更简单的“直接交流”视图，不强行套任务结构。

已落地部分：

- request_summary 已新增 Subtask View，可显示 plan 中每一步的描述、期望结果以及 completed / active / blocked / pending 状态。
- lite_chat 请求在无任务计划时会回退为更简单的 Direct exchange 视图，不强行展示多步骤任务结构。

后续落点：

- Subtask View 应优先作为 Textual 右侧或底部的独立组件落地，而不是只在 /request_summary 文本输出中存在。
- lite_chat 和 task 两类请求应在组件层有不同密度的展示，而不是只靠文案区分。

### 4.3.4 工具执行反馈卡片

当前状态：部分完成。

- 工具调用时显示工具名、参数摘要、执行状态、结果摘要。
- 工具失败时给出重试建议或替代动作，而不是只把失败埋在 checkpoint 里。

已落地部分：

- request_summary 已新增 Tool Feedback 视图，可汇总 detached tools 与最近一次 triage 中的工具失败信号，展示 tool、status、summary、source、run_id。
- 阻塞类请求现在可以直接看到对应工具的错误类型与 ask_user 之类的动作建议，不必只从 failure details 文本里硬读。

后续落点：

- Tool Feedback 应从 request_summary 的附属文本，迁成 Textual 中可持续可见的状态组件。
- detached tools、tool failure、reroute 提示应合并进同一块运行态反馈区域，减少信息散落。

## 4.4 Phase 4: 高效执行命令与自动辅助

目标：让终端真正帮助用户完成动作，而不是只展示状态；并把高频动作逐步从命令输入迁到组件操作。

### 4.4.1 常用动作预设

当前状态：部分完成。

- 将一批高频工作流组合成快捷命令。
- 例如：查看最近失败请求、查看失败详情、恢复最近阻塞请求、查看最近 detached tools。
- 预设动作应尽量围绕“排障”和“继续工作”设计。

已落地部分：

- 已新增 /latest_failure [summary|snapshots|resume]，可直接打开最近一次 failed 或 blocked request 的详情、快照或恢复动作。
- 已新增 /resume_last_blocked，可一键对最近的 blocked request 执行 reroute 恢复。
- 已新增 /detached_tools [request_id]，可快速查看指定请求或最近请求的 detached tools。

### 4.4.2 上下文动作入口

当前状态：部分完成。

- 根据当前上下文提供少量明确的恢复入口和继续动作。
- 让列表结果、失败恢复、快照恢复形成顺滑的下一步，而不是要求用户记住额外提示系统。
- 避免在终端中额外维护一层“推荐语义”，减少噪音。

已落地部分：

- 已新增 /latest_failure [summary|snapshots|resume]、/resume_last_blocked、/detached_tools [request_id] 等上下文动作入口。
- 已支持 /pick、/selection、/clear_selection，让“先查看，再继续动作”成为单条链路。
- recommend 行与 /recommend 命令已移除，避免终端界面持续打断主交流流。

后续落点：

- Action Context 组件应逐步承接这些高频动作入口，至少支持查看摘要、查看快照、恢复阻塞、查看 detached tools 等快捷触发。
- commands.py 继续保留动作后端，但不应继续把“上下文动作体验”主要建设在纯命令帮助文案上。

### 4.4.3 输入辅助

当前状态：部分完成。

- 支持命令补全、参数提示、最近命令历史。
- 支持对 request_id、snapshot 名称、MCP 名称等做候选补全。
- 降低记忆成本与输入错误率。

已落地部分：

- 已新增 /suggest [prefix]，可对命令、recent requests、最近请求的 snapshot 名称和最近输入做候选提示。
- 已新增 /recent_commands [limit]，可回看当前 CLI 会话内最近输入的命令或查询，便于复用。
- 行式 CLI 下未直接接 Tab 补全；切到 Textual 后，可在单一输入控件上继续收口候选补全与快捷操作。

### 4.4.4 命令与自然语言协同

当前状态：部分完成。

- 除显式命令外，允许部分自然语言请求自动映射为内部命令。
- 例如“看一下最近失败请求”“恢复上一个阻塞任务”可以直接转换成 CLI 动作。
- 这部分需要谨慎设计边界，避免误触发破坏性操作。

已落地部分：

- 已新增安全映射，仅覆盖少量明确、无破坏性的自然语言动作，例如“看一下最近失败请求”“恢复上一个阻塞任务”“查看对话历史”。
- 映射命中时会先回显 Mapped to command，再执行内部命令，避免动作来源不透明。
- 当前未开放有副作用或歧义较高的自然语言操作，优先控制误触发风险。

## 5. 建议优先级

如果按当前收益排序，建议优先顺序如下：

1. Textual 组件骨架与固定区域继续收口。
2. Action Context、Recent History、Current Request 从文本摘要升级为可操作组件。
3. 子任务视图与工具反馈视图迁入 Textual 固定区域。
4. recent/request_summary/list_snapshots/resume 的联动操作从命令串联升级为组件选择。
5. 输入补全、候选提示和自然语言映射与 Textual 输入框整合。
6. 旧行式 CLI 保持回退可用，但不再作为主体验建设方向。

## 6. 第一阶段建议直接落地的事项

为了尽快改善日常使用体验，建议先落地下面这批直接作用于 Textual 工作台的小切口：

### 6.1 输出表现

当前状态：已完成。

- 给用户输入后新增“意图重述”展示行。
- 给 lite mode 请求新增轻量交流标识。
- 给 agent_blocked 输出新增“建议下一步”行。
- 给 request_summary 输出新增更紧凑的人类可读摘要。

### 6.2 命令易用性

当前状态：已完成。

- 为 /request_summary、/recent_requests、/request_rollup、/list_snapshots、/resume_snapshot 增加短别名。
- 允许 /request_summary、/list_snapshots 默认使用 last_request_id。
- 增加“恢复最近阻塞请求”“查看最近失败请求”的快捷命令。

### 6.3 交互连续性

当前状态：已完成。

- recent_requests 结果增加序号选择模式。
- request_summary 结果中追加“可继续动作”提示。
- snapshots 列表中支持直接输入序号恢复。

## 7. 验收标准

当后续终端改造达到以下状态时，可以认为方向基本正确：

- 用户在不翻帮助的情况下，也能完成最常见的查看、恢复、继续执行操作。
- 一次请求的关键信息可以在终端主视图中直接读懂，不需要频繁切换命令。
- 普通交流、轻聊天、任务执行、阻塞恢复四类场景在视觉和交互上明显区分。
- 高频命令输入长度明显下降，错误率下降。
- 新用户第一次进入 CLI 时，就能从界面中看出“现在能做什么、下一步做什么”。

## 8. 结论

llm_brain 下一阶段最值得投入的，不是继续增加更多底层能力，而是把已有能力组织成一个更顺手、更清楚、更适合长期使用的终端工作界面。

在路径上，这件事现在已经不再主要落在 app/cli/main.py 和 app/cli/terminal_ui.py，而是应该优先落在 main.py -> app/cli/textual_app.py 这条主入口链上。

先把“看得清、用得顺、继续得下去”解决好，再把这些能力沉淀成稳定组件，后面的调度、恢复、记忆、技能、观测能力才会真正发挥价值。
