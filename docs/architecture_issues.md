# llm_brain 架构问题清单 / 待修复项

## 1. 文档目的

本文档基于当前代码实现整理项目的架构问题、工程风险与待修复项，用于后续开发排期、重构评估和问题闭环。这里记录的是“当前代码中真实存在的问题”，不是对目标态能力的抽象设想。

## 2. 问题总览

当前项目的主要问题集中在五类：

1. 主执行链虽然可运行，但记忆、技能、MCP 等系统没有真正闭环。
2. 多个接口处于占位或半实现状态，文档目标与代码现状存在落差。
3. 工程层面的会话管理和工具路由仍较粗糙，不利于扩展。
4. 依赖与结果持久化还不够稳定，不利于后续演化。
5. 仍存在少量实现层面的可维护性问题。

## 3. 已完成修复

以下问题已在当前轮修复完成，不再列为待办主项：

### 3.1 路径配置化

已完成内容：

- `memory/memory_manager.py` 不再依赖硬编码绝对路径。
- `skills_md/skill_parser.py` 不再依赖硬编码绝对路径。
- `config.json` 与 `config.py` 新增并接管 `skills_md_dir`、`memory_db_path`、`memory_backup_dir`。

当前状态：

- 路径现在按项目根目录解析，仓库可移动性明显提升。

### 3.2 Markdown 技能自动加载

已完成内容：

- `SkillManager` 初始化时会自动扫描并加载 `skills_md/` 目录中的 `.md` 技能。
- 支持刷新缓存并避免重复加载同一文件。

当前状态：

- Markdown 技能系统已经形成最小启动闭环。

### 3.3 CLI 动态技能加载

已完成内容：

- `/load_skill` 不再是占位。
- 当前支持按文件名加载 `.py` 或 `.md` 技能文件。

当前状态：

- CLI 已具备最小运行时技能加载能力。

### 3.4 明显实现瑕疵修复

已完成内容：

- `skills_md/skill_parser.py` 已补充 `Optional` 导入。
- `memory/memory_manager.py` 的 `load_full_memory()` 已增加空结果保护。

### 3.5 主执行链闭环修复

已完成内容：

- `agent_core.py` 已将记忆检索正式接入主流程：`initial_planning` 先做摘要召回，`call_model_subtask` 会将摘要结果注入上下文，并在候选不足时补充完整记忆内容。
- 记忆系统已形成“会话级记录 + 步骤级记录”的最小建模：初始请求先落一条会话记忆，子任务执行后追加步骤记忆，最终输出与反思摘要会回写到会话主记录。
- 会话 ID 不再硬编码为 `session_1`。`AgentCore` 会生成动态会话 ID，CLI 启动时自动创建会话，并支持通过 `/new_session` 切换新会话。
- `select_active_tools()` 已增加基于工具名称和描述的关键词过滤，避免默认把全部 Python 工具塞进上下文。
- `Reflector` 的 `retry` 分支已增加最大重试次数保护，超过阈值后会升级为阻断并提示用户介入。

当前状态：

- 原 4.1、4.2、4.3、5.2、5.3 已完成修复，不再列为当前待办主项。

### 3.6 MCP 最小链路落地

已完成内容：

- `mcp_servers/mcp_manager.py` 新增最小 MCP 配置适配层，支持读取 JSON/YAML 配置并生成 LangChain `StructuredTool`。
- `agent_core.py` 的 `_auto_load_mcp_servers()` 现在会自动扫描 `mcp_servers/` 目录并注册配置中的工具。
- `load_mcp_server()` 不再是占位输出，已支持按文件名或绝对路径加载 MCP 配置，并将工具并入 Agent 可调用工具集。
- 仓库新增 `mcp_servers/demo_mcp.json` 作为最小可运行示例配置。

当前状态：

- 5.1 已从“纯占位”提升为“配置式最小接入”，后续主要待补的是真实 MCP 连接与资源生命周期管理。

### 3.7 依赖清单可维护性修复

- 依赖清单已重建为标准 UTF-8 文本，可直接查看、编辑和安装。

当前状态：

- 原 6.1 已完成修复，不再列为待办问题。

### 3.8 技能注册与路由统一

已完成内容：

- `skills_md/skill_parser.py` 已扩展为统一的能力注册中心，除 Markdown 技能外，也会接收 Python 工具和 MCP 工具的能力元数据。
- `agent_core.py` 在加载 Python 技能和 MCP 工具时，会同步将它们注册到 `SkillManager`。
- 子任务执行阶段现已由 `SkillManager.assign_capabilities_to_task()` 统一返回 Prompt 技能和候选工具，Markdown 与 Python/MCP 不再走完全割裂的选择链路。

当前状态：

## 5. 中优先级问题

### 5.1 MCP 集成尚未接入真实协议连接

- `agent_core.py`
- `mcp_servers/`

问题描述：

- 当前 MCP 仍是“配置驱动的本地适配层”，不是标准 MCP 协议客户端。
- 现阶段会把配置文件中的工具声明转成 LangChain 工具，但不会建立真实远端连接。
- 资源发现、连接保活、断连恢复等生命周期逻辑尚未实现。

影响：

- 当前已经具备最小工具注册能力，但与真实 MCP 服务的兼容性仍未验证。
- 若后续直接接标准 MCP server，还需要补 transport、handshake 和资源同步机制。

建议修复：

1. 在当前配置式工具注册基础上，引入真实 transport 定义和握手流程。
2. 区分“本地适配 MCP 配置”和“标准 MCP server 连接”两类模式。
3. 增加资源注册、连接状态和失败回退策略。

注：原 5.2、5.3 已在上一轮修复完成；原 5.1 的“最小链路落地”已完成，当前中优先级主项变为真实 MCP 协议接入。

## 7. 文档与实现的认知偏差风险

### 7.1 目标态文档容易被误读为现状实现

受影响位置：

- `docs/requirements.md`
- `docs/agent.md`

问题描述：

- 需求文档描述了丰富的认知、记忆、技能演化、可信度体系等目标。
- 但当前代码只实现了其中一部分骨架。

影响：

- 后续接手者容易高估完成度。
- 排查问题时可能误以为某些能力已经存在。

建议修复：

1. 持续维护 [project_overview.md](./project_overview.md)，作为“源码事实总览”。
2. 对目标态文档补充“当前实现状态”标记，或在 README 中持续强调区别。

## 8. 推荐修复顺序

建议按以下顺序推进：

1. 先把记忆检索和结果回写正式接入主执行链路。
2. 然后为会话 ID、重试次数、工具筛选建立状态管理。
3. 再推进 MCP 集成和更复杂的可信度体系。
4. 期间同步收敛技能载体格式与依赖版本策略。

## 9. 结论

当前项目最核心的问题不是“没有架构方向”，而是“架构方向已经明确，但很多关键子系统还停在半连接状态”。后续修复工作的重点，应优先放在把已有模块真正接上主流程，而不是继续扩展新的概念层设计。只有先补齐路径、会话、技能加载、记忆读写这些基础链路，后续的反思、技能演化和 MCP 扩展才有稳定落点。
